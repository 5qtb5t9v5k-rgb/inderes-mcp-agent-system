## Detail per finding

### 1. Comparison routing systematically too thin 🔴

**The data:** 19 `is_comparison=true` runs got *only* `["quant"]`. Another 10
got `["quant","research"]`. Of the 30+ comparison queries in the dataset,
**only 1** got the full {quant,research,sentiment,valuation} route.

**Examples (run_id):**
- `2026-05-09T13:21` — *"Vertaile Sammon ja Nordean kannattavuutta"* → `["quant"]`
- `2026-05-08T20:13` — *"nordea vs. aktia"* → `["quant"]`
- `2026-05-07T16:17` — *"nordea vs aktia"* → `["quant"]`
- `2026-05-08T20:59` — *"Miten Nordean ROE-kehitys eroaa muista pohjoismaisista pankeista?"* → `["quant","valuation"]` (no research!)

**Why it matters:** profitability comparisons need RESEARCH (operating
context, business-model differences) and SENTIMENT (analyst view per
company) to be more than a number-spreadsheet. Currently LEAD compensates
by *speculating* about business models — that's hallucination territory.

**Eval case sketch:**
```yaml
case_001_comparison_routing:
  query: "Vertaile Sammon ja Nordean kannattavuutta"
  expect_router_domains_min: [quant, research, sentiment]
  expect_router_companies: [Sampo, Nordea]
  expect_router_is_comparison: true
```

**Probable root cause:** router prompt doesn't have a strong "comparison
queries always need ≥ 3 domains" rule. Quick fix candidate.

### 2. Päättely structured form is dead 🟠

**The data:** 96 `paattely_kind = empty` vs 55 `prose`. **Zero structured.**

Spec says LEAD should emit:
```json
{ "disagree": "...", "resolution": "...", "uncertain": "...", "skipped": "..." }
```

In practice it always emits `{ "prose": "..." }` or fails entirely.

**Why it matters:** the structured form was the basis of the 2×2 slot grid
in the UI. Right now the UI falls back to prose every time, so the slot
grid is dead code. Either fix the prompt or drop the structured branch.

**Eval case sketch:**
```yaml
case_002_paattely_schema:
  query: "tee arvonmääritys Sampolle"  # multi-domain, conflict-prone
  expect_paattely_kind: structured     # OR: drop the test if we choose to deprecate
```

### 3. Conflict-detector under-firing 🟠

**The data:** of 119 multi-agent eligible runs:
- 32 found ≥1 conflict (27 %)
- 54 found ≥1 agreement (45 %)
- 65 found neither (55 %)

**Why it matters:** the detector is a separate LLM call (~$0.005 / run, ~3 s
latency added). If it produces nothing on 55 % of runs, the cost / value
ratio is poor. Either:
- The detector's prompt is too strict (only fires on clear conflicts)
- LEAD's combined input format isn't parseable by the detector
- Single-domain runs slip through the multi-agent gate (we filter `subagent_count >= 2` but maybe still too thin)

**Eval case sketch:**
```yaml
case_003_conflict_coverage:
  query: "tee arvonmääritys Bittiumille manual override ROE 5%"
  expect_conflict_signals_min: 1   # this case has known conflicts
```

### 4. `search-companies` 9.7 % error rate 🟠

**The data:** of 176 `search-companies` calls, 17 errored.
- Other tools: `get-fundamentals` 0 %, `get-inderes-estimates` 1.6 %,
  `list-content` 0 %, etc.

**Why it matters:** `search-companies` is the entry tool — typo-tolerance
matters. `Vincit`, `Witted Megacorp`, certain small caps may not be in
the MCP catalog. We need fuzzy matching or a graceful "not found" message
that LEAD can surface ("yhtiötä ei löytynyt MCP-tietokannasta").

### 5. Noise (empty/trivial/0-subagent) 🟡

12 empty-query + 20 trivial + 23 zero-subagent. Cleanup: filter these out
of any eval set (`WHERE has_query = 1 AND is_trivial_query = 0 AND subagent_count > 0`).
The indexer already flags them.

### 6. Reproducibility unverified 🟡

Top repeats:
- `kerro puuilon myymälämäärän muutoksesta...` — 7×
- `tee arvonmääritys nordealle` — 6× in 24 h
- `nokia` — 6×

**Eval case sketch:**
```yaml
case_005_reproducibility:
  query: "tee arvonmääritys Nordealle"
  run_n_times: 3
  expect_structural_consistency:
    - same domains routed
    - same fair_value within ±5 %
    - same Inderes recommendation
```

A delta on any of these is a model-drift signal that warrants a regression flag.

### 7. Latency outliers 🟡

- `Vincitin nykytilanteesta`: 308 s with 2 subagent errors — Vincit not
  in catalog, tools fail repeatedly.
- `selitä mistä Nordean kannattavuus oikeasti tulee`: 239 s + 70 s on two
  runs — single-domain (`research,quant`) deep dives. RESEARCH may be
  doing many sequential `read-document-sections` calls without a hard cap.

**Recommendation:** hard limit `max_tool_calls_per_subagent = 12` in the
already-planned hard-limits item (BACKLOG §0 Wk 1).

## Pro-tier signal (preliminary, n=2)

Pro tier was used in only 2 of 183 runs. Compared to Flash Lite:
- Pro avg duration: 27.1 s vs 21.1 s (28 % slower — expected)
- Pro avg synthesis chars: 1418 vs 2138 (34 % shorter — surprising)
- Pro warnings_surfaced: 0/2 vs 17/149 (too small to compare)

**Caveat:** sample size 2 is meaningless for quality assessment. Need ≥20
matched-query Pro vs Flash Lite pairs to draw any conclusion.

## Next-up eval cases (golden.yaml seed)

Based on these findings, the first 6 cases for `evals/golden.yaml` write
themselves:

1. **Comparison routing** — `case_001_comparison_routing`
2. **Päättely schema** — `case_002_paattely_schema` (or drop)
3. **Conflict-detector firing** — `case_003_conflict_coverage`
4. **Tool typo tolerance** — `case_004_search_robustness` (use a known-bad
   company name, expect graceful failure not silent error)
5. **Reproducibility** — `case_005_reproducibility`
6. **Latency cap** — `case_006_latency_cap` (run a query, verify
   `duration_s < 60` and `tool_calls < 12 per subagent`)

These six cover all the systemic issues found in 183 real runs. Each is
testable by re-running the query and asserting against the indexer
output — no LLM-judge needed for Tier 0 assertions, only for Tier 1
qualitative grading.

## Tier-2 (Supabase) hooking

When we move from SQLite to Supabase Postgres:
- Same schema works directly (Postgres dialect compat).
- The `runs` and `tool_calls` tables become canonical, not derived.
- `evals_judgments`, `evals_cases`, `evals_proposals` are the new
  Supabase-only tables (not in the SQLite indexer because we don't
  have judgments yet).
- The indexer becomes a *streaming* process (write to Supabase as the
  pipeline runs) instead of a batch rebuild. ~1 day to refactor.
